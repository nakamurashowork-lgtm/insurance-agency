# t_case_comment 拡張方針

## 1. 問題

`t_case_comment` は `target_type`（`renewal_case` / `accident_case`）と
対応するNULL許容のIDカラム（`renewal_case_id` / `accident_case_id`）を持つポリモーフィック設計である。

この構造は案件種別が増えるたびにカラム追加が必要になるため、
将来的に営業案件コメントや顧客コメントが必要になった場合の拡張コストが高い。

## 2. 方針決定

### 2-1. 現フェーズの方針

現行の `t_case_comment` 構造を**維持する**。

理由は以下のとおり。

- 現在の対象は `renewal_case` と `accident_case` の2種のみであり、CHECK制約で整合性が保たれている
- Phase C 以降で営業案件コメントが必要になる時点で改めて検討する
- 構造変更は既存データのマイグレーションを伴うため、必要性が確定した時点で着手する

### 2-2. 拡張が必要になる条件

以下のいずれかが発生した時点で本方針を再評価し、DDL変更を検討する。

- 営業案件（`t_sales_case`）へのコメント機能が要件になった場合
- 顧客直接へのコメント機能が要件になった場合
- 3種類目以上の案件コメントが必要になった場合

### 2-3. 将来拡張時の設計選択肢

拡張が必要になった場合の選択肢を以下に整理する。

**選択肢A：カラム追加方式（現行の延長）**

`sales_case_id` カラムと対応する CHECK 制約を追加する。
シンプルだが案件種別ごとにカラムが増え続ける。
3種程度までなら許容範囲。

**選択肢B：エンティティ参照の汎化**

`entity_type`（VARCHAR）と `entity_id`（BIGINT）の2カラムで対象を汎化する。
案件種別の増加に対して構造変更が不要になる。
ただし、外部キー制約をDB側では張れないため整合性はアプリ層で保証する必要がある。

**推奨**：拡張対象が3種以上になる場合は選択肢Bへの移行を検討する。
移行時は既存レコードの `renewal_case_id` / `accident_case_id` を
`entity_type` / `entity_id` に変換するマイグレーションが必要になる。

## 3. 現時点での対応

現行 DDL の変更は行わない。

本方針書を `docs/` に配置し、拡張判断の基準として参照する。
営業案件コメントの要件が具体化した時点で、本文書をアップデートする。
