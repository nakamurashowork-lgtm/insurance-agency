# フォルダ構成方針

## 1. 目的

この文書は、`docs/` を仕様の正、`config/ddl/` をスキーマの正としたときの、リポジトリ構成ルールを定義する。

実装都合で仕様を変えることを防ぎ、保守時にどこを見れば正しいかを即座に判断できる状態を維持することを目的とする。

## 2. 正本の定義

- 仕様の正: `docs/`
- スキーマの正: `config/ddl/common/`, `config/ddl/tenant/`

`docs/` 配下では、上位方針（例: 画面マップ）と個別画面仕様に加え、全画面共通の UI 規約（`docs/06_common-ui-rules.md`）を正式仕様として扱う。

上記以外の説明文、コメント、補助SQLは参考情報であり、正本ではない。

## 3. ルート配下の責務

- `docs/`: 正式仕様（上位設計、画面設計）
- `config/ddl/`: canonical DDL（変更は設計合意の後にのみ実施）
- `public/`: Web入口と画面エンドポイント
- `src/`: ドメイン、アプリケーション、インフラ実装
- `config/`: 実行設定、接続設定、バッチ設定
- `tmp/`: 一時生成物専用（コミット対象外を推奨）
- `tools/`: 開発補助スクリプト（運用導線と分離）

## 4. public 配下の整理ルール

- 通常導線の正式画面は、`docs/03_screen-map.md` と一致させる
- 非正規導線、旧画面、暫定機能は通常導線から分離する
- 旧画面を残す場合は、少なくとも次を満たす
  - main nav に出さない
  - ホームの日常業務に出さない
  - 権限で一般利用者導線から外す

## 5. 命名と参照の整合ルール

- テーブル名は canonical DDL と一致させる
- 実装コードで DDL 未定義テーブルを参照しない
- 画面名、機能名、用語は `docs/` の記述を優先して統一する

## 6. 変更時の手順

1. `docs/` の該当仕様を確認する
2. `config/ddl/` との整合を確認する
3. 差分がある場合は、実装変更前に差分を明示する
4. 実装を修正する
5. 影響範囲（画面、API、バッチ、権限）を記録する

## 7. 削除判断の基準

次のいずれかに該当するファイルは削除候補とする。

- `docs/` と `config/ddl/` の正本に照らして役割が重複している
- 旧導線専用で、正式導線から到達しない
- 一時作業用で再現性がない
- 空ディレクトリや未使用補助資産

ただし、削除前に参照元調査を行い、運用影響がないことを確認する。

## 8. この文書の優先順位

フォルダ構成の運用判断で迷った場合は、次の優先順位で判断する。

1. `docs/` の仕様
2. `config/ddl/` の canonical DDL
3. 実装コード

実装と仕様が衝突した場合は、実装を正とせず、差分として扱って是正する。
