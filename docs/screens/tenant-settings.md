# テナント設定

## 1. 画面概要

### 1-1. 画面名

テナント設定

### 1-2. 画面ID

SCR-TENANT-SETTINGS

### 1-3. 目的

テナント単位で利用する各種設定を管理し、通知動作や業務で利用する選択肢マスタを、運用に合わせて変更できるようにする。

### 1-4. 画面の位置づけ

テナント単位の運用設定を扱う管理画面。  
画面はタブを使わず、通知設定・マスタ管理・SJNETコード設定を 1 画面内で縦積み表示する。

通常業務の主導線には含めず、管理者向けの補助導線またはヘッダーメニュー配下から利用する。  
ホームから遷移させる場合も、管理・設定として扱う。

---

## 2. 基本方針

* テナント単位の設定変更を行う画面とする。
* 画面は以下の 3 セクションで構成する。
  * テナント概要
  * 通知設定
  * マスタ管理（担当者・対応状況・種目・活動目的・ユーザー表示名）
* 同一画面内を縦スクロールして操作し、独立画面には分けない。
* 設定系画面であり、通常業務の主要入口にはしない。
* 主利用者は管理権限を持つ利用者とする。
* 初期段階で参照範囲を広めに取る場合でも、管理画面である位置づけは変えない。
* テナントをまたぐ設定変更はできない。
* 保存対象はテナント単位で明確に分かるようにする。

---

## 3. 想定利用者

* テナント管理者
* 管理者ユーザー
* system_admin 相当の利用者

※ 一般のテナント所属ユーザーは、本画面の主利用者とはしない。  
※ 初期実装で参照可能とする場合でも、通常業務の標準利用画面とは扱わない。

---

## 4. 主な利用シナリオ

### 4-1. 通知設定変更シナリオ

1. 管理権限を持つ利用者がテナント設定を開く。
2. 通知設定セクションを確認する。
3. 満期通知や事故リマインドの通知条件を確認する。
4. 必要に応じて通知日数や通知先設定を変更する。
5. 保存後、テナント単位の通知設定として反映する。

### 4-2. マスタ管理シナリオ

1. 管理権限を持つ利用者がテナント設定を開く。
2. 画面下部へスクロールしてマスタ管理セクションへ移動する。
3. 業務で使用する選択肢マスタの一覧を確認する。
4. 必要に応じてマスタ値を追加、更新、削除する。
5. 保存後、対象テナントの業務画面に反映する。

### 4-3. SJNETコード設定シナリオ

1. 管理権限を持つ利用者がテナント設定を開く。
2. SJNETコード設定セクションへ移動する。
3. SJ-NETの代理店コードと担当者の対応一覧を確認する。
4. 未登録の代理店コードを新たに追加する、または既存の対応を更新・無効化する。
5. 保存後、次回のSJNET取込から担当者が自動設定されるようになる。

## 6. 通知設定セクション

### 6-1. 目的

満期通知や事故リマインドなど、テナント単位で制御したい通知設定を管理する。

### 6-2. 想定設定項目

| 項目 | 内容 |
| --- | --- |
| 満期通知有効フラグ | 満期通知を有効にするか |
| 満期通知開始日数 | 満期日の何日前から通知するか |
| 満期通知間近日数 | 間近通知を何日前で扱うか |
| 事故リマインド有効フラグ | 事故リマインドを有効にするか |
| 通知先種別 | LINE WORKS などの通知先種別 |
| 通知先情報 | Webhook URL 等の通知先情報 |
| 通知本文設定 | 必要に応じて定型文や件名ルール |

※ 実際の保持項目に応じて調整する。

### 6-3. 主操作

| 操作 | 内容 |
| --- | --- |
| 保存 | 通知設定を保存する |
| 入力値リセット | 変更前の状態へ戻す |

### 6-4. 動作

* 初期表示時に対象テナントの通知設定を表示する。
* 利用者が値を変更して保存すると、対象テナントの設定を更新する。
* 保存成功時はメッセージを表示する。
* 入力不正がある場合は保存しない。

### 6-5. 表示上の考え方

* 満期通知と事故通知をグループ分けして表示する。
* 通知先設定は誤入力しやすいため、ラベルを明確にする。
* 有効フラグがオフの場合、関連項目を非活性にしてよい。

## 7. マスタ管理セクション

### 7-1. 目的

テナントごとに管理したい業務選択肢を、画面から追加・更新・削除できるようにする。

### 7-2. 想定管理対象

| マスタ種別 | 内容 |
| --- | --- |
| 種目 | 保険種目などの選択肢 |
| 営業担当者 | 営業担当の選択肢 |
| 事務担当者 | 事務担当の選択肢 |
| 支店 | 支店の選択肢 |
| ステータス候補 | 一覧や編集で使う状態選択肢 |

※ 実際に管理対象とするマスタは実装仕様に合わせて調整する。

### 7-3. 画面構成

マスタ管理セクション内は、管理対象ごとに一覧と編集領域を持つ構成とする。

| 項目 | 内容 |
| --- | --- |
| マスタ種別選択 | 管理したいマスタ種別を選ぶ |
| 一覧表示 | 登録済みマスタ値を一覧表示する |
| 追加入力欄 | 新規値を追加する |
| 編集導線 | 既存値を編集する |
| 削除導線 | 既存値を削除する |
| 表示順 | 必要に応じて並び順を管理する |
| 有効フラグ | 必要に応じて利用可否を管理する |

### 7-4. 主操作

| 操作 | 内容 |
| --- | --- |
| 追加 | 新しいマスタ値を登録する |
| 更新 | 既存のマスタ値を更新する |
| 無効化 | 既存のマスタ値を無効化する（物理削除しない） |
| 有効化 | 無効化済みのマスタ値を再度有効化する |
| 並び順変更 | 表示順を変更する |

### 7-5. 動作

* 初期表示時に選択中マスタ種別の一覧を表示する。
* 追加・更新・無効化・有効化後は一覧を再表示する。
* マスタ値は物理削除せず `is_active = 0` による無効化を行う。無効化したマスタ値は新規選択肢に表示されなくなるが、既存データの表示には影響しない。
* `is_system = 1` のマスタ値（固定）は無効化・更新できない。
* 無効化できない条件がある場合はメッセージ表示する。

### 7-6. 表示上の考え方

* 管理対象が切り替わっても操作方法が大きく変わらないようにする。
* 業務利用頻度の高いマスタは上位に置く。
* 有効フラグ運用がある場合、物理削除より無効化を優先してよい。

## 8. SJNETコード設定

### 8-1. 目的

SJ-NETの満期一覧データに含まれる「代理店コード」と、本システムの担当者（`m_staff`）を紐づける。  
この設定がないと、SJNET取込時に契約・満期案件の担当者が自動設定されない。

**初回SJNET取込前に、全代理店コードの登録を完了させることを強く推奨する。**

### 8-2. 代理店コードについて

SJ-NETの満期一覧データには「代理店コード」（8桁、例: N8559000）が含まれており、担当者1人に1コードが割り当てられている。このコードはSJ-NET上の識別子であり、システムのスタッフIDとは独立している。

### 8-3. 管理場所

SJNETコードは独立セクションではなく、**マスタ管理セクションの「担当者」（`m_staff`）に統合**して管理する。

担当者マスタの各レコードに `sjnet_code` フィールドがあり、そこに代理店コードを入力することで紐づけが完成する。

| m_staff フィールド | 内容 |
| --- | --- |
| staff_name | 担当者名（システム上の表示名） |
| sjnet_code | SJNETの代理店コード（例: N8559000）。UNIQUE KEY |
| is_sales | 営業担当者フラグ |
| is_office | 事務担当者フラグ |
| is_active | 有効フラグ。OFFの場合は取込時に担当者未設定となる |

### 8-4. 主操作

担当者マスタの追加・編集フォームで `sjnet_code` を入力する。  
詳細はマスタ管理セクション（§7）を参照。

### 8-5. SJNET取込との連携

SJNET取込時に `m_staff.sjnet_code` を参照し、以下を自動設定する。

| 設定先 | フィールド | 動作 |
| --- | --- | --- |
| t_contract | sales_staff_id | 解決した m_staff.id。既に設定済みの場合は上書きしない |
| t_renewal_case | assigned_staff_id | 解決した m_staff.id。既に設定済みの場合は上書きしない |

代理店コードが未登録または `is_active = 0` の場合、担当者は設定されず取込は続行する（エラー扱いにしない）。  
取込結果に「マッピング未登録: N件」として表示される。

### 8-6. 表示上の考え方

* 担当者マスタ一覧に `sjnet_code` 列を表示し、設定済み・未設定を一目で把握できるようにする。
* 代理店コードはそのまま表示し、変換・省略しない。
* SJNET取込結果に `unresolved` 件数が出た場合、担当者マスタの `sjnet_code` 入力を促すメッセージを表示する。

## 10. 画面動作

### 10-1. 初期表示

* ログイン中ユーザーの所属テナントに対応する設定を表示する。
* 画面上部からテナント概要、通知設定、マスタ管理の順で表示する。
* テナント名を明示し、どのテナント設定を編集しているか分かるようにする。

### 10-2. セクション移動時

* 画面遷移や表示切替は行わず、スクロールで上下のセクションを確認する。
* 未保存変更がある場合の扱いは実装方針に従う。

### 10-3. 通知設定保存時

* 入力チェック後、対象テナントの通知設定を保存する。
* 保存成功時は成功メッセージを表示する。
* 保存失敗時はエラーメッセージを表示する。

### 10-4. マスタ追加・更新・無効化・有効化時

* 入力チェック後、対象テナントのマスタを更新する。
* 処理後は一覧を再取得して表示する。
* マスタ値は物理削除せず `is_active = 0` による無効化で対応する。
* `is_system = 1`（固定）の場合は無効化できない旨をメッセージ表示する。

### 10-5. 担当者マスタのsjnet_code設定時

* `sjnet_code` の重複チェックを行う（UNIQUE KEY）。
* `is_active = 0` に設定すると取込時の自動解決から除外される。
* 保存成功時は一覧を再表示する。

## 11. 本画面の役割整理

### 11-1. テナント設定の役割

本画面の役割は、テナント単位で業務運用に関わる設定を変更できるようにすることである。

ここで扱うものは以下である。

* 通知の有効・無効
* 通知条件
* 通知先情報
* 業務で使う選択肢マスタ
* SJNETの代理店コードとユーザーの対応

### 11-2. ホームとの違い

ホームは状況確認と導線の画面である。  
本画面は、運用設定そのものを変更する管理画面である。

### 11-3. 分割画面にしない理由

通知設定・マスタ管理・SJNETコード設定はどれもテナント単位の設定であり、利用者視点では同じカテゴリの操作であるため、同一画面内を縦スクロールで連続して扱う。

## 14. 権限・認可

* ログイン済みユーザーのみ表示可能とする。
* 表示・更新対象は所属テナントの設定に限定する。
* 本画面は管理機能として扱い、編集権限は管理権限者に限定することを基本とする。
* 初期実装で参照のみを広く許容する場合でも、更新権限は管理権限者に限定することが望ましい。
* 編集権限がない場合は参照のみ、または画面自体を非表示としてよい。
* 権限差分は通常業務の主導線ではなく、管理機能の公開範囲として扱う。

## 16. 例外・考慮事項

### 16-1. 通知先情報が未設定の場合

* 未設定状態を明示する。
* 通知有効化時に必須入力チェックを行ってよい。

### 16-2. マスタ値が他画面で利用中の場合

* 物理削除は行わず、`is_active = 0` による無効化を行う。無効化したマスタ値は新規選択肢には表示しないが、既存データの値はそのまま保持・表示する。

### 16-3. 複数テナント所属の場合

* 初期仕様では主所属テナントのみ編集対象とする。
* 将来的に切替が必要になった場合は別途検討する。

### 16-4. 未保存状態で画面を離れる場合

* 必要に応じて確認ダイアログを表示してよい。

### 16-5. SJNETコードが未登録のまま取込を実行した場合

* 取込はエラーにならず続行する。
* 取込結果に未解決コードと件数を表示し、本セクションへの誘導リンクを表示する。
* 担当者未設定の契約・満期案件は、満期案件一覧の担当者フィルタで「未設定」として確認できる。

* テナントをまたぐ更新が起きないこと。

* PC とスマホの両方で操作できること。

## 19. ユーザー管理タブ（2026-04-04 追加）

### 概要

テナント所属ユーザーの業務表示名（`users.display_name`）を管理者が変更できるセクション。  
管理者のみアクセス可能。通常のプロフィール設定はユーザー自身では変更できない。

### 表示項目

| 列 | 内容 |
|---|---|
| 業務表示名 | `display_name`。NULL の場合は「（Google名）」を付記して `name` を表示 |
| Googleアカウント名 | `users.name`（読み取り専用・参考表示） |
| メールアドレス | `users.email`（読み取り専用・参考表示） |
| ロール | `user_tenants.role`（読み取り専用・参考表示） |

### 操作

- 各行の「編集」ボタンでダイアログを開く
- `display_name` を入力して保存
- 空欄で保存すると `display_name = NULL` となり、Googleアカウント名にフォールバック

### データソース

- `common.users.display_name`（変更対象）
- `common.user_tenants`（テナント所属確認・ロール取得）

### 権限

管理者ロール（`admin`）またはシステム管理者のみ。

## 20. 目標管理セクション（2026-04-07 追加）

### 20-1. 目的

年度ごとの保険料合計目標（`target_type = 'premium_total'`）を登録・編集する。  
月次目標および他の目標種別（損保・生保・件数）は本セクションでは扱わない。

詳細仕様は `docs/policies/12_sales-target-spec.md` を参照。

### 20-2. 構成

| 項目 | 内容 |
|---|---|
| 年度選択 | プルダウン。前年・当年・来年の3年分（URL クエリ `target_fy` で保持） |
| チーム全体目標 | `staff_user_id IS NULL` の目標。1件のみ |
| 担当者別目標 | テナント所属ユーザーごとに1件ずつ |

### 20-3. 主操作

| 操作 | 内容 |
|---|---|
| 年度切替 | プルダウン変更で即時リロード。URL クエリ `target_fy` を更新 |
| 保存 | 行ごとに個別保存（UPSERT 動作）。`POST tenant/sales-target/save` |
| 削除 | チーム全体目標のみ削除可能。`POST tenant/sales-target/delete` |

### 20-4. 画面動作

- 保存成功時: 「目標を保存しました」メッセージ + 同じ年度の画面を再表示
- 削除成功時: 「目標を削除しました」メッセージ + 同じ年度の画面を再表示
- 担当者別目標は削除不可。0 を保存することで実質的に目標未設定と同等になる
- 削除ボタン押下時はアプリ内確認ダイアログを表示する

### 20-5. 権限

管理者ロール（`admin`）またはシステム管理者のみ保存・削除操作が可能（既存の権限制御を流用）。
