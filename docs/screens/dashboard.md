# ホーム

## 1. 画面概要

ホームは、ログイン後に最初に表示される入口画面である。
本画面の役割は、利用者がその日に確認すべき業務をすばやく把握し、必要な一覧画面へ迷わず進めるようにすることにある。

ホームは便利だからといって、何でもここで完結させてはいけない。
ここはあくまで **全体状況の把握と業務への入口** に徹する。
実際の検索、確認、更新、記録は各一覧画面および詳細画面で行う。

## 2. 画面の目的

本画面の目的は以下の3点である。

* 今日見るべき業務をひと目で分かるようにすること
* 優先して確認すべき対象へすぐ進めること
* 業務ごとの正式画面への入口をまとめること

ホームで大事なのは、情報量の多さではない。
利用者が「次にどこへ行けばよいか」を迷わないことの方が重要である。

## 3. 想定利用者

* 営業担当
* 内勤担当
* 管理者

営業担当は当日の対応対象を確認するために使う。
内勤担当は契約、顧客、事故、成績の入口として使う。
管理者は全体の滞留や件数感をざっと確認するために使う。

## 4. 画面の位置づけ

ホームは、各業務画面の上に立つ集約画面である。
ただし、各業務の正本画面ではない。

役割分担は明確にする。

* ホーム: 状況把握と入口
* 満期一覧: 契約を探す画面
* 満期詳細: 契約を処理する画面
* 顧客一覧: 顧客を探す画面
* 顧客詳細: 顧客全体を確認する画面
* 事故案件一覧 / 事故案件詳細: 事故対応を進める画面
* 成績管理一覧: 成績を管理する画面

特に重要なのは、**満期一覧が契約一覧を兼ね、満期詳細が契約詳細を兼ねる** という点である。
ホームから契約専用画面へは遷移させない。

## 5. 主な導線

### 5-1. 日常業務の主導線

```text
ログイン
  ↓
ホーム
  ↓
満期一覧
  ↓
満期詳細
```

もっとも利用頻度の高い導線である。
ホームは、満期業務の対象を見つけるための入口となる。

### 5-2. 見込管理の導線

```text
ホーム
  ↓
見込案件一覧
  ↓
見込案件詳細
```

見込案件の件数を確認し、対象案件へ進む導線である。

### 5-3. 事故業務の導線

```text
ホーム
  ↓
事故案件一覧
  ↓
事故案件詳細
```

### 5-4. 成績業務の導線

```text
ホーム
  ↓
成績管理一覧
```

### 5-5. 設定業務の導線

```text
ホーム
  ↓
テナント設定
```

## 6. 画面構成

ホームは、利用者が上から順に見れば自然に優先順位が分かる構成とする。

### 6-1. ヘッダ情報領域

利用者名やテナント名など、現在の利用文脈が分かる情報を表示する。

想定項目:

* 利用者名
* 所属またはテナント名
* 現在日時

### 6-2. 重点確認エリア（要確認）

その日に優先して見るべき要確認情報のみを表示する。

想定表示:

* 満期: 対応遅れ件数 / 7日以内件数
* 事故: 高優先度未完了件数

**このエリアは常に全体（全担当者）の件数を表示する。**
業務入口カードの「自分|全体」トグルとは独立しており、トグルの影響を受けない。

ここで件数を見せる目的は、分析ではなく優先順位づけである。
グラフを飾りとして置くより、押すべき入口が分かるほうが重要だ。

補足:

* 満期の「今日期限」は上段ではなく中段カードで表示する
* 上段は警報性の高い2指標に絞る

### 6-3. 業務別ショートカットエリア（業務入口）

各正式画面への入口をまとめて表示する。3枚のカードで構成する。

| カード | 表示内容 | リンク先 |
|--------|----------|----------|
| 満期業務（未完了） | 14日以内 / 28日以内 / 60日以内 件数 | 満期一覧 |
| 事故案件（未完了） | 高優先度 / 中優先度 / 低優先度 件数 | 事故案件一覧 |
| 見込管理 | 見込A / 見込B / 今月成約予定 件数 | 見込案件一覧 |

**担当者選択ドロップダウンについて**

* 各カードのタイトル行右側に担当者選択ドロップダウンを配置する
* ドロップダウンは3枚のカードで独立して動作する
* 選択肢:
  1. 自分（ログインユーザー名を併記）
  2. 全体
  3. テナント所属の他ユーザー一覧（display_name 昇順）
* 初期値は「自分」とする
* 「自分」: 担当者（assigned_staff_id）がログインユーザーに紐づく件数
* 「全体」: テナント内全件
* 他ユーザー選択時: 担当者がそのユーザーに紐づく件数
* 要確認エリアはこのドロップダウンの影響を受けず、常に全体を表示する

権限制御は行わない。全ログインユーザーが他ユーザーの数字を参照できる。

顧客一覧はナビバーに常時表示されているため、業務入口カードには含めない。
「顧客を探す」は辞書的な行動であり、毎日の業務入口には馴染まない。

管理・設定導線は通常業務導線と分離し、下段に権限者のみ表示する。

### 6-4. 補助サマリエリア（下部）

成績サマリの下に、補助情報カードを横並びで配置する。

| カード | 表示内容 |
|--------|----------|
| 営業活動 | 今日の活動件数 / 日報提出状態（常にログインユーザー自身） |
| 成績管理 | 今月計上件数 / 今月保険料（成績管理一覧への入口） |

営業活動カードはドロップダウンを持たず、常にログインユーザー自身の値を表示する。
成績管理カードのスコープは成績サマリの担当者選択ドロップダウンと連動する。

## 7. 主な表示内容

### 7-1. 満期業務サマリ

ホームで最優先に見せる対象である。

想定内容:

* 今日期限件数（中段カード）
* 近日満期件数（中段カード）
* 対応遅れ件数（上段警報）

このサマリから満期一覧へ遷移する。
ここで個別契約の編集までは行わない。

### 7-2. 見込管理サマリ

見込案件の状況を簡易表示し、見込案件一覧へ導く。

想定内容:

* 見込A / 見込B の件数（未完了ステータスのみ）
* 今月成約予定の件数（expected_contract_month = 当月）

集計対象は `t_sales_case` の `is_deleted = 0` かつ未完了ステータス（won/lost 以外）。

### 7-3. 事故業務サマリ

事故案件の滞留や未完了状況を簡易表示し、事故案件一覧へ導く。

### 7-4. 成績業務サマリ

成績管理一覧への入口を表示する。

成績サマリは **年度累計** を単独カードとして表示する（「今月」カードは持たない）。

| 表示項目 | 内容 |
|----------|------|
| 年度累計（千円） | 4月〜当月の保険料合計 |
| 前年同期累計 | 前年度同期間の保険料合計 |
| 前年比 | 年度累計 ÷ 前年同期累計 × 100（%） |
| 年度目標 | 担当者の年度目標（未設定時は「目標未設定」） |

担当者選択ドロップダウンおよび年度切り替えプルダウンと連動する。

## 8. 主な操作

### 8-1. 表示時

* 利用者向けのサマリ情報を表示する
* 各業務への入口を表示する
* 優先度の高い件数を目立つ位置に出す

### 8-2. 遷移操作

* 満期一覧へ遷移する
* 事故案件一覧へ遷移する
* 見込案件一覧へ遷移する
* 活動一覧へ遷移する
* 成績管理一覧へ遷移する

顧客一覧へはナビバーから遷移する（ホームの業務入口カードからは遷移しない）。

管理者のみ:

* 管理・設定へ遷移する

### 8-3. 設けない操作

* ホーム上で契約内容を直接更新する
* ホーム上で満期対応を直接完了させる
* ホーム上で顧客詳細や事故詳細を編集する

ホームに処理機能を持たせ始めると、責任範囲が崩れて必ず散らかる。
ここは入口に徹した方がいい。

## 9. UI上の考え方

### 9-1. 上から順に優先順位が分かること

利用者は細かい説明を読まない。
そのため、画面上部に重要な件数、その下に業務入口、その下に補助情報という順に並べる。

### 9-2. 件数は分析ではなく行動のために出すこと

数字を並べるだけでは意味がない。
見るべき件数と押すべき導線がつながって初めて意味がある。

### 9-3. 情報を盛りすぎないこと

ホームに全部載せたい気持ちは分かるが、それをやるとただのごちゃついた一覧になる。
必要なのは、今の仕事を始めるための最小限の整理である。

## 10. 満期画面系との関係

ホームは契約業務の入口だが、契約の正本画面ではない。
契約に関する正式な一覧・詳細は以下とする。

* 満期一覧 = 契約一覧
* 満期詳細 = 契約詳細

この前提を崩して、ホームから契約専用画面を別に作る設計にはしない。
ここをブレさせると、設計全体がまた割れる。

## 11. 顧客画面系との関係

顧客確認が必要な場合、ホームは顧客一覧への入口だけを持つ。
顧客の全体像確認は顧客詳細で行う。

ホームで顧客情報を深掘りし始めると、顧客一覧・顧客詳細の存在意義が薄れ、画面分担が壊れる。

## 12. 権限の考え方

ホームに表示する件数や入口は、利用者権限に応じて制御できるようにする。

例:

* 一般利用者: 上段（満期/事故）、中段（満期/事故/顧客/成績）を表示
* 管理者: 上記に加えて下段の管理・設定導線を表示
* 事故案件導線は通常業務として一般利用者にも表示

見えてはいけない件数や業務入口を全員に見せる設計にはしない。

## 13. エラー時の考え方

一部サマリの取得に失敗した場合でも、画面全体を使えなくしない。
取得できた情報は表示し、失敗した領域だけ理由を示す。

例:

* 満期件数の取得失敗
* 事故件数の取得失敗
* 権限不足により非表示

ホームは入口画面なので、全部こけたら仕事が止まる。ここは壊れにくく作るべきだ。

## 14. 今回の設計で確定すること

* ホームは入口画面であり、処理画面ではない
* 上段（要確認）は「満期: 対応遅れ/7日以内」「事故: 高優先度未完了」に絞って表示する
* 要確認エリアは常に全体（全担当者）の件数を表示する。担当者選択ドロップダウンの影響を受けない
* 業務入口カードは「満期業務・事故案件・見込管理」の3枚構成とする
* 顧客一覧への導線はナビバーに委ね、業務入口カードには含めない
* 業務入口カード（満期業務・事故案件・見込管理）と成績サマリの各カードに
  担当者選択ドロップダウンを持つ
* ドロップダウンの選択肢は「全体」（先頭）とテナント所属ユーザーのフラットな一覧とする（「自分」特別エントリは設けない）
* 初期値は「全体」とする
* 営業活動カードはドロップダウンを持たず、常にログインユーザー自身の値を表示する
* 担当者選択は権限制御なし。全ログインユーザーが他ユーザーの数字を参照できる
* 営業活動カードは業務入口エリアではなく、成績サマリ下の補助エリアに配置する
* 満期一覧が契約一覧を兼ねる
* 満期詳細が契約詳細を兼ねる
* 契約専用画面への独立導線は作らない
* 事故案件一覧への導線は日常業務として正式に持つ
* 見込案件一覧への導線は業務入口カードから正式に持つ
* 成績サマリは「年度累計」単独カードとして表示する（「今月」カードは持たない）
* 成績サマリの年度切り替えはプルダウン（`<select>`）とし、年度表記は西暦（YYYY年度）に統一する

## 15. 補足

本ドキュメントはホーム画面の役割、構成、導線、表示方針を定めるものである。
項目の正式名称、表示条件、件数ロジック、権限別表示制御、レイアウト詳細、API仕様などの詳細は、別途詳細設計で定義する。

## 16. 禁止事項

ホームは入口画面であるため、以下を置いてはならない。

- 登録・更新・削除のフォームまたはボタン
- 一覧テーブルへの「追加」行（例：「＋ 活動を追加」行）
- 詳細画面を経由しない直接編集操作

これらが必要な場合は、対応する正式画面（各一覧・詳細画面）に配置する。  
`page-subtitle` への説明文記載も禁止する（`docs/policies/06_common-ui-rules.md` 付録B 参照）。
