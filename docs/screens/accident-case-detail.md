# 事故案件詳細

## 1. 画面概要

### 1-1. 画面名

事故案件詳細

### 1-2. 画面ID

SCR-ACCIDENT-CASE-DETAIL

### 1-3. 目的

事故案件の詳細情報を確認し、案件の文脈を踏まえて、更新・対応状況確認・コメント確認・監査ログ確認を行えるようにする。

### 1-4. 画面の位置づけ

事故案件一覧から遷移する詳細確認画面。  
案件単位で内容を確認し、更新、コメント確認、監査ログ確認を行うための画面とする。

本画面は事故案件機能の詳細画面ではあるが、現時点では通常運用の正式な主要導線とはしない。  
管理者向け補助導線または限定利用の残存機能の一部として扱う。

---

## 2. 基本方針

* 一覧画面では不足する案件文脈を、本画面で確認できるようにする。
* 更新は本画面から行うことを基本とする。
* コメント、監査ログは本画面内で確認する補助情報とし、独立画面として扱わない。
* 本画面は、事故対応の状況判断に必要な情報をまとめて確認するためのハブとする。
* 一覧画面に判断責務を持たせず、詳細画面で対応経過を確認する前提とする。
* 本画面は満期案件詳細と同格の通常主導線とは扱わない。
* 削除が必要な場合でも例外操作として扱い、本画面の中心責務にはしない。

---

## 3. 想定利用者

* 管理者ユーザー
* 事故案件機能の利用を許可されたテナント所属ユーザー

※ 一般利用者向けの通常業務画面とはしない。

---

## 4. 主な利用シナリオ

### 4-1. 基本シナリオ

1. 利用者が事故案件一覧から対象案件の詳細を開く。
2. 本画面で事故案件の基本情報、対応状況、担当者情報を確認する。
3. コメントや監査ログを確認し、過去の対応経緯を把握する。
4. 必要であれば更新を行う。

### 4-2. 対応経過確認シナリオ

* 利用者が、現在どこまで対応済みか、誰が更新したかを確認したい場合に本画面を開く。
* コメントや監査ログを見て、案件の流れを把握する。

### 4-3. 更新判断シナリオ

* 利用者が、状態変更や担当者修正などの更新を行う前に、本画面で現状を確認する。
* 事実確認の後に編集へ進むことを基本とする。

---

## 5. 画面レイアウト

### 5-1. 全体構成

画面は以下のブロックで構成する。

1. ページヘッダー
2. 案件基本情報エリア
3. 対応状況エリア
4. コメントエリア
5. 監査ログエリア

### 5-2. ページヘッダー

| 項目 | 内容 |
| --- | --- |
| 画面タイトル | 事故案件詳細 |
| 一覧へ戻る | 事故案件一覧へ戻る |
| 編集ボタン | 対象案件を更新する |
| 削除導線 | 必要な場合のみ利用する例外操作 |

### 5-3. 案件基本情報エリア

事故案件そのものの基本情報を表示する。

| 項目 | 内容 |
| --- | --- |
| 事故日 | accident_date |
| 顧客名 | customer_name |
| 種目 | product_type |
| 支店 | branch |
| 担当者 | staff |
| 備考 | 必要に応じて補足情報 |

※ 実際の項目は現行データ構造に合わせて調整する。

### 5-4. 対応状況エリア

案件の進捗や対応判断に関わる情報を表示する。

| 項目 | 内容 |
| --- | --- |
| ステータス | 未対応、対応中、完了 など |
| 最終更新日時 | 最後に更新された日時 |
| 最終更新者 | 最後に更新したユーザー |
| 進捗把握用メモ | 必要に応じて表示 |

### 5-5. コメントエリア

案件対応上の補足メモや経緯を確認するための領域。

| 項目 | 内容 |
| --- | --- |
| コメント一覧 | 登録済みコメントの一覧 |
| コメント入力欄 | 新規コメント入力 |
| コメント登録ボタン | コメント保存 |

### 5-6. 監査ログエリア

案件に対する更新履歴を確認するための領域。

| 項目 | 内容 |
| --- | --- |
| 更新日時 | 変更が行われた日時 |
| 更新者 | 変更したユーザー |
| 変更項目 | 変更された項目名 |
| 変更前 | 変更前の値 |
| 変更後 | 変更後の値 |

監査ログは本画面内の補助情報として表示し、独立画面としては扱わない。

## 6. 主操作

| 操作 | 内容 | 優先度 |
| --- | --- | --- |
| 編集 | 事故案件を更新する | 主操作 |
| コメント登録 | コメントを追加する | 補助操作 |
| 一覧へ戻る | 一覧へ戻る | 補助操作 |

※ 削除は例外操作であり、通常の主操作には含めない。

---

## 7. 画面動作

### 7-1. 初期表示

* 一覧画面で選択した案件IDに対応する詳細情報を取得して表示する。
* 基本情報、対応状況、コメント、監査ログを表示する。
* データ取得に成功した範囲で画面を構成する。

### 7-2. 編集押下時

* 対象案件の編集画面または編集モードへ遷移する。
* 詳細確認後に編集へ進む流れを基本とする。

### 7-3. コメント登録時

* 入力内容を保存する。
* 保存後はコメント一覧を更新して表示する。
* 登録失敗時はエラーメッセージを表示する。

### 7-4. 監査ログ表示

* 取得済みの監査ログを時系列で表示する。
* 件数が多い場合は一部表示または折りたたみ表示を検討してよい。

### 7-5. 一覧へ戻る押下時

* 事故案件一覧へ戻る。
* 一覧の検索条件や表示位置は可能な範囲で保持することが望ましい。

---

## 8. 本画面で扱う情報の役割整理

### 8-1. 事故案件詳細の役割

本画面は、案件単位で状況を把握するための中心画面である。

ここで確認すべきものは以下である。

* 今どの案件を見ているか
* その案件の基本情報は何か
* 現在のステータスはどうか
* 誰が担当しているか
* 過去にどのような更新があったか
* 補足コメントに何が残っているか

### 8-2. コメントの役割

コメントは、事故対応上の補足情報を残すための補助情報である。  
本画面の中で文脈付きで確認する前提とする。

### 8-3. 監査ログの役割

監査ログは独立画面ではなく、本画面の補助情報である。  
案件の更新経緯を確認するためのものであり、事故案件詳細の文脈の中で見る前提とする。

## 12. 権限・認可

* ログイン済みユーザーのみ表示可能とする。
* 表示対象は所属テナントの案件に限定する。
* 本画面は通常導線の主要画面ではなく、権限を持つ利用者に限定して表示してよい。
* 編集、削除、コメント登録は権限に応じて可否を制御してよい。
* 監査ログ表示も必要に応じて権限制御対象としてよい。

---

## 13. スマホ表示方針

* 情報ブロックを縦積みで表示する。
* 主要操作である編集ボタンと一覧へ戻る導線は上部で見える位置に置く。
* コメントと監査ログは折りたたみ表示を許容する。

---

## 14. 例外・考慮事項

### 14-1. コメントが存在しない場合

* コメントなしと表示する。
* エラー扱いにはしない。

### 14-2. 監査ログが存在しない場合

* 監査ログなしと表示する。
* エラー扱いにはしない。

### 14-3. 監査ログ件数が多い場合

* 最新分を優先表示する。
* 全件を常時展開しない。

### 14-4. 一部案件情報が欠落している場合

* 取得できた項目のみ表示する。
* 欠落項目は空欄または未設定として表示してよい。
* 画面全体をエラーにしない。

### 14-5. 通常導線に含めない方針との関係

* 本画面は事故案件機能の詳細画面ではあるが、現時点では通常業務の正式主導線とは扱わない。
* 満期案件詳細と同格の通常中心画面として扱わない。

---

## 15. 非機能要件上の留意点

* 詳細画面に一覧相当の情報量を詰め込まない。
* コメントや監査ログが多くても、主要情報の視認性を損なわない構成とする。
* データ取得失敗時に画面全体が利用不能にならないよう配慮する。
* PC とスマホの両方で操作できること。
