# 成績管理

## 1. 画面概要

### 1-1. 画面名

成績管理

### 1-2. 画面ID

SCR-SALES-PERFORMANCE-LIST

### 1-3. 目的

成績データを検索、絞り込みし、対象を特定して成績詳細へ進める。

### 1-4. 画面の位置づけ

成績管理機能の一覧入口画面。  
本画面は「探す」責務に集中し、1件確認と更新は成績詳細で行う。

満期業務とは別軸の主要入口であり、元データ探索の起点として扱う。

---

## 2. 基本方針

* 一覧画面は検索、絞り込み、対象特定に集中する。
* 一覧の主導線は「詳細を開く」とする。
* 新規登録とCSV取込は一覧上部の補助操作として扱う。
* 編集と削除は成績詳細画面起点を基本とする。
* 一覧に全情報を詰め込まず、主要項目のみ表示する。
* 1件の確認は成績詳細へ寄せる。
* 成績の集計や分析はホーム側でも一部表示するが、元データ確認は一覧と詳細で分担する。

## 3. 想定利用者

* 代理店担当者
* 管理者ユーザー
* テナント所属ユーザー

## 4. 主な利用シナリオ

### 4-1. 基本シナリオ

1. 利用者が成績管理を開く。
2. 検索条件で対象成績を絞り込む。
3. 一覧で対象成績を特定する。
4. 対象行から成績詳細を開く。
5. 成績詳細で内容確認し、必要に応じて編集または削除を行う。
6. 一括反映が必要な場合は一覧からCSV取込を使う。

### 4-2. 新規登録シナリオ

* 利用者が成績追加ボタンを押下する。
* 登録モーダルを開き、必要項目を入力して保存する。
* 保存成功後、一覧へ反映する。

### 4-3. 詳細確認シナリオ

* 利用者が一覧の詳細導線から対象成績を開く。
* 成績詳細画面で省略項目を含めて内容を確認する。
* 必要時に編集、削除へ進む。

### 4-4. CSV取込シナリオ

* 利用者がCSV取込ボタンを押下し、CSV取込ダイアログを開く。
* 成績管理簿から出力したCSVファイルを選択する。
* 取込実行ボタンを押下する。
* **1行に損保と生保が混在している行は、自動的に損保1件・生保1件の2レコードに分割して登録する。**
* 取込完了後、ダイアログ内に取込結果サマリとエラー行一覧を表示する。
* エラー行がある場合は内容を確認し、顧客登録または手動登録で対応する。

## 5. 一覧画面共通UIルール

本画面は満期一覧で定めた共通ルールに準拠して実装する。

### 5-1. 骨格

* 画面タイトルと主操作ボタン
* 折りたたみ可能な検索条件カード
* 一覧カード（上部サマリ + テーブル + 下部ページャー）
* これらが同じ主コンテナ幅に統一される

### 5-2. ページング

* 一覧上部と下部の両方にページャーを配置
* 表示件数切替は 10 / 50 / 100
* 件数表示例: 「120件中 1-10件を表示」
* 条件変更時は先頭ページへ戻す

### 5-3. ソートと状態保持

* ヘッダクリックでソート可能
* 検索条件、ページ、表示件数、ソート条件をURLで保持
* 一覧から詳細へ進み、詳細から戻っても同じ状態を維持

### 5-4. 主操作の配置

* 新規登録（成績を追加）とCSV取込は画面タイトル下のアクションエリアに配置
* 一覧行の操作は「詳細を開く」を主導線にする

## 8. 画面動作

### 8-1. 初期表示

* 初期表示時は既定条件に基づく成績一覧を表示する。
* 検索条件は初期値を設定してよい。
* 一覧はデフォルトソート順で表示する。

### 8-2. 検索実行時

* 入力条件に一致する成績データを再取得して表示する。
* 検索結果件数を表示する。
* 該当データがない場合は0件として表示する。

### 8-3. 成績追加押下時

* 成績登録モーダルを開く。
* 入力値を保存後、一覧を再表示する。
* 保存成功時は成功メッセージを表示する。

### 8-4. 詳細押下時

* 対象行の成績詳細画面へ遷移する。
* 一覧では省略した項目を詳細画面で確認する。
* 編集、削除は詳細画面から実行する。

### 8-5. CSV取込押下時

1. CSV取込ダイアログを開く。
2. 利用者がCSVファイルを選択する。
3. 取込実行ボタンを押下する。
4. 処理中はプログレス表示またはローディング表示を行う。
5. 処理完了後、ダイアログ内に取込結果パネルを表示する。
   * 成功件数・エラー件数をサマリ表示する。
   * エラー行がある場合はエラー行一覧を表示する。
6. エラー行がある場合、利用者は内容を確認して対応する（詳細は セクション18 参照）。

### 8-6. データが存在しない場合

* 一覧部にデータなしを表示する。
* エラー扱いにはしない。

## 11. 共通一覧UI実装ルール

成績管理一覧は、満期一覧と同じ一覧UIルールに準拠する。

* 検索条件は折りたたみ表示とし、検索実行後も開閉状態を保持する
* 検索、表示件数変更、並び替えは route パラメータを維持して遷移する
* 表示件数は 10 / 50 / 100 の3択とする
* ページャーは一覧上部・下部に表示し、現在件数レンジを明示する
* ヘッダーソート対象は「計上日 / 契約者名 / 種目 / 保険料 / 精算月」とする
* 一覧から詳細遷移時は検索条件・ページ・表示件数・並び順をURLに保持する
* 詳細から一覧へ戻る時も同じ状態を維持する

## 15. 例外・考慮事項

### 15-1. 一覧件数が多い場合

* ページングを行う。
* 必要に応じて検索条件の利用を前提とする。

### 15-2. 保険料など数値項目の表示

* 桁区切りを行い、視認性を確保する。
* **マイナス保険料（解約返戻・等級訂正）は赤字で表示する。**
* 通貨単位の扱いは実装仕様に従う。

### 15-3. CSV取込件数が多い場合

* 取込処理中の待機状態を利用者に分かるようにする。
* 完了後は件数や結果を表示することが望ましい。

### 15-4. 一部項目が未設定の場合

* 空欄または未設定として表示する。
* 画面全体をエラーにしない。

## 18. CSV取込仕様

### 18-1. 対象ファイル

成績管理簿（Excel）から出力したCSVファイルを対象とする。  
文字コードは UTF-8 または Shift-JIS（BOM付き）を許容する。  
1行目・2行目はヘッダー行としてスキップする。

### 18-2. CSVカラム定義

全26列。損保部分（列1〜19）と生保部分（列20〜26）が1行に混在する形式をそのまま受け入れる。

**損保部分（列1〜19）**

| 列番号 | フィールド名 | 内容 | 備考 |
| --- | --- | --- | --- |
| 1 | performance_year | 計上年（和暦。例: 7 = 令和7年） | 必須 |
| 2 | performance_month | 計上月 | 必須 |
| 3 | performance_day | 計上日 | 必須 |
| 4 | receipt_no | 領収証番号 | 空欄可 |
| 5 | customer_name | 損保契約者名 | 損保レコード生成の必須判定に使用 |
| 6 | policy_start_year | 始期年（和暦） | 空欄可 |
| 7 | policy_start_month | 始期月 | 空欄可 |
| 8 | policy_start_day | 始期日 | 空欄可 |
| 9 | is_auto | 自動車チェック（○ or 空） | |
| 10 | is_fire | 火災チェック（○ or 空） | |
| 11 | is_other | その他チェック（○ or 空） | |
| 12 | settlement_month_raw | 精算月 | 空欄可 |
| 13 | premium_amount | 損保保険料（マイナス可） | 損保レコード生成の必須判定に使用 |
| 14 | （月次累計） | 取込不要 | スキップ |
| 15 | （損保挙積累計） | 取込不要 | スキップ |
| 16 | installment_seq | 分割払い回数（例: 5 = 5回払い） | 空欄可 |
| 17 | installment_num | 分割回目（例: 1 = 1回目） | 空欄可 |
| 18 | product_type | 種目（自動車・火災・SGP等） | 空欄可 |
| 19 | remark | 損保備考 | 空欄可 |

**生保部分（列20〜26）**

| 列番号 | フィールド名 | 内容 | 備考 |
| --- | --- | --- | --- |
| 20 | （区切り列） | 取込不要 | スキップ |
| 21 | life_policy_start_date | 生保始期日（YYYY/MM/DD） | 空欄可 |
| 22 | life_application_date | 生保申込日（YYYY/MM/DD） | performance_dateとして使用 |
| 23 | life_customer_name | 生保契約者名 | 生保レコード生成の必須判定に使用 |
| 24 | life_product_type | 生保種目（がん・医療・変額・収入等） | |
| 25 | life_premium_amount | 生保保険料（マイナス可） | 生保レコード生成の必須判定に使用 |
| 26 | life_remark | 生保備考（限定告知等） | 空欄可 |

### 18-3. 分割ルール

**1行のCSVデータから最大2レコードを独立して登録する。**

損保と生保は完全に独立したレコードとして扱う。一方のレコード登録に失敗しても、他方はロールバックしない。

```
【損保レコード生成判定】
  customer_name（列5）が空でない
  AND premium_amount（列13）が空でない
  → 損保レコードを1件生成する（source_type = 'non_life'）

【生保レコード生成判定】
  life_customer_name（列23）が空でない
  AND life_premium_amount（列25）が空でない
  → 生保レコードを1件生成する（source_type = 'life'）
  → 損保レコードとは独立して処理する

【スキップ判定】
  customer_name と premium_amount がともに空
  → 空行または累計行とみなしてスキップ（エラー扱いにしない）
```

### 18-4. 損保レコードへのカラムマッピング

| t_sales_performance カラム | ソース列 | 変換ルール |
| --- | --- | --- |
| `performance_date` | 列1・2・3 | 和暦→西暦変換（令和: 列1 + 2018）。DATE型に組み立て |
| `performance_type` | （自動判定） | `premium_amount < 0` の場合 `cancel_deduction`、それ以外は `new` |
| `source_type` | （固定） | `'non_life'` |
| `customer_id` | 列5 | 顧客名でm_customerを検索して解決（後述） |
| `policy_start_date` | 列6・7・8 | 和暦→西暦変換。すべて空の場合はNULL |
| `insurance_category` | 列9〜11 | `is_auto=○` → `'自動車'`、`is_fire=○` → `'火災'`、`is_other=○` → `'その他'`。複数チェックは先頭を採用し備考に付記 |
| `product_type` | 列18 | そのまま |
| `premium_amount` | 列13 | 数値変換。マイナス値をそのまま許容 |
| `installment_count` | 列16 | 値があれば設定。空の場合はNULL |
| `settlement_month` | 列12 | `YYYY-MM` 形式に正規化。空の場合はNULL |
| `receipt_no` | 列4 | 空の場合はNULL |
| `remark` | 列19 | そのまま |
| `application_date` | （なし） | NULL（損保には申込日概念なし） |

### 18-5. 生保レコードへのカラムマッピング

| t_sales_performance カラム | ソース列 | 変換ルール |
| --- | --- | --- |
| `performance_date` | 列22 → 列21 | 生保申込日（列22）を優先。空の場合は始期日（列21）を使用。両方空の場合はエラー（`missing_date`） |
| `performance_type` | （自動判定） | `life_premium_amount < 0` の場合 `cancel_deduction`、それ以外は `new` |
| `source_type` | （固定） | `'life'` |
| `customer_id` | 列23 | **損保の customer_id とは独立して**、生保契約者名でm_customerを別途検索して解決 |
| `policy_start_date` | 列21 | そのまま。空の場合はNULL |
| `application_date` | 列22 | そのまま。空の場合はNULL |
| `insurance_category` | （固定） | `'生命保険'` |
| `product_type` | 列24 | そのまま |
| `premium_amount` | 列25 | 数値変換。マイナス値をそのまま許容 |
| `remark` | 列26 | そのまま |
| `settlement_month` | （なし） | NULL（生保には精算月概念なし） |
| `receipt_no` | （なし） | NULL |
| `installment_count` | （なし） | NULL |

### 18-6. 和暦→西暦変換ルール

```
令和の場合: 西暦年 = 和暦年 + 2018
  例: 令和7年 → 7 + 2018 = 2025
  例: 令和8年 → 8 + 2018 = 2026

日付の組み立て:
  DATE(和暦年 + 2018, 月, 日)

エッジケース:
  - 年・月・日がすべて空 → NULL（始期日はNULL許容）
  - 年のみあって月日が空 → エラー（invalid_date）
  - 不正な日付（2月30日等）→ エラー（invalid_date）
```

### 18-7. customer_id の解決ルール

損保・生保それぞれで、契約者名から `m_customer.id` を解決する。

```
1. customer_name で m_customer.customer_name を完全一致検索
2. 1件ヒット → そのcustomer_idを使用して登録
3. 0件ヒット → エラー種別 missing_customer としてエラー行バッファに追加
4. 2件以上ヒット → エラー種別 ambiguous_customer としてエラー行バッファに追加
```

生保契約者名は損保契約者名と独立して解決する。  
生保の customer_id 解決に失敗した場合、生保レコードのみをエラー扱いとし、損保レコードは正常登録する。

### 18-8. staff_user_id の解決ルール

損保・生保それぞれのレコードに対して、SJNET担当者コードから `staff_user_id` を解決する。

解決には `m_sjnet_staff_mapping` テーブルを使用する。

```
1. CSV の担当者コード（拠点コード + 担当者コード）で m_sjnet_staff_mapping を検索
2. 1件ヒット → マッピング先の staff_user_id を使用して登録
3. 0件ヒット → staff_user_id = NULL で登録し、エラー種別 unmapped_staff として
              エラー行バッファに追加（登録は続行する）
4. is_active = 0 のマッピング → 0件ヒットと同様に扱う
```

`staff_user_id` が解決できない場合でも、レコードの登録は続行する。  
`unmapped_staff` エラーは登録失敗ではなく警告として扱い、取込結果パネルに一覧表示する。

担当者マッピングの登録・管理はテナント設定のSJNETコード設定セクションで行う。  
マッピング未登録が多い場合は、取込結果からテナント設定へ直接遷移できる導線を表示することが望ましい。

### 18-9. エラー分類

| エラー種別 | 判定条件 | 取込結果での表示 |
| --- | --- | --- |
| `missing_customer` | 契約者名でm_customer検索結果が0件 | 未登録の顧客名・行番号を表示。顧客登録または手動登録を促す |
| `ambiguous_customer` | 契約者名でm_customer検索結果が2件以上 | 行番号・候補顧客名を表示。名寄せまたは手動登録を促す |
| `unmapped_staff` | 担当者コードがm_sjnet_staff_mappingに未登録 | 行番号・担当者コードを表示。テナント設定でSJNETコードを登録するよう促す |
| `missing_date` | 生保レコードで申込日・始期日がともに空 | 行番号・生保契約者名を表示。手動登録を促す |
| `invalid_date` | 日付として不正な値（年のみ・存在しない日付等） | 行番号・不正値・列名を表示 |
| `invalid_amount` | premium_amount が数値に変換できない | 行番号・不正値・列名を表示 |
| `missing_required` | customer_name と premium_amount がともに空でない行で必須項目が欠落 | 行番号を表示してスキップ |

`missing_customer` および `ambiguous_customer` はレコード登録不可のため、対象行はスキップする。  
`unmapped_staff` および `missing_date`（生保）は登録を試行するが、取込結果に警告として記録する。

### 18-11. 取込フロー

```
CSVファイル選択
  ↓
ヘッダー行スキップ（1〜2行目）
  ↓
行ループ
  ├─ 空行・累計行スキップ
  │    （customer_name と premium_amount がともに空の行）
  │
  ├─ 【損保処理】
  │    必須判定（customer_name・premium_amount が空でないか）
  │      → NG: missing_required エラー行に追加
  │    顧客名解決
  │      → 0件: missing_customer エラー行に追加
  │      → 複数件: ambiguous_customer エラー行に追加
  │      → 1件: 担当者コード解決へ
  │    担当者コード解決（m_sjnet_staff_mapping）
  │      → 未登録: unmapped_staff 警告を追加、staff_user_id = NULL で続行
  │      → 解決済: 損保レコード INSERT
  │
  └─ 【生保処理】（life_customer_name が空でない場合のみ）
       必須判定（life_customer_name・life_premium_amount が空でないか）
         → NG: missing_required エラー行に追加
       日付確認（申込日・始期日のどちらかが存在するか）
         → 両方空: missing_date エラー行に追加
       顧客名解決（損保とは独立）
         → 0件: missing_customer エラー行に追加
         → 複数件: ambiguous_customer エラー行に追加
         → 1件: 担当者コード解決へ
       担当者コード解決（m_sjnet_staff_mapping）
         → 未登録: unmapped_staff 警告を追加、staff_user_id = NULL で続行
         → 解決済: 生保レコード INSERT

全行処理完了
  ↓
取込結果パネルをダイアログ内に表示
  ├─ サマリ（損保登録数・生保登録数・エラー行数）
  └─ エラー行一覧（エラー種別・行番号・契約者名・対応方法）
  ※ unmapped_staff が1件以上ある場合はテナント設定へのリンクを表示
```
