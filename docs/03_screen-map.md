# 画面マップ

## 1. 目的

本システムの画面構成と主要導線を整理する。

画面は増やしすぎず、営業担当が日常業務で迷わず使えることを優先する。
特に、満期業務では **満期一覧 = 契約一覧**、**満期詳細 = 契約詳細** として扱い、契約専用の一覧画面・詳細画面は別途設けない。

また、顧客軸での確認や連絡履歴の把握が必要なため、顧客一覧・顧客詳細は正式画面として持つ。

## 2. 画面一覧

### 2-1. 正式画面

| 画面ID                | 画面名     | 役割                                           |
| ------------------- | ------- | -------------------------------------------- |
| SCR-LOGIN           | ログイン    | メールアドレス、パスワード、二段階認証により利用者を認証する               |
| SCR-DASHBOARD       | ダッシュボード | 今日見るべき案件、満期対象、事故対応、実績入力などの入口をまとめる            |
| SCR-RENEWAL-LIST    | 満期一覧    | 契約一覧を兼ねる。満期日、担当者、契約者、証券番号などで契約を探し、対応対象を絞り込む  |
| SCR-RENEWAL-DETAIL  | 満期詳細    | 契約詳細を兼ねる。契約内容、満期対応、対応履歴、メモ、関連顧客情報を一体で確認・更新する |
| SCR-CUSTOMER-LIST   | 顧客一覧    | 顧客名、電話番号、担当者などで顧客を検索し、対象顧客を特定する              |
| SCR-CUSTOMER-DETAIL | 顧客詳細    | 顧客の基本情報、連絡先、契約一覧、対応履歴を確認する                   |
| SCR-SALES-LIST      | 実績管理一覧  | 実績を検索、登録、編集、削除する                             |
| SCR-ACCIDENT-LIST   | 事故案件一覧  | 事故対応中の案件を一覧で確認し、優先度順に処理する                    |
| SCR-ACCIDENT-DETAIL | 事故案件詳細  | 事故概要、進捗、対応履歴、メモを確認・更新する                      |
| SCR-TENANT-SETTINGS | テナント設定  | 通知設定、各種マスタ設定など、テナント単位の運用設定を管理する              |

### 2-2. 設けない画面

以下は独立画面としては持たない。

| 画面名  | 理由              |
| ---- | --------------- |
| 契約一覧 | 満期一覧が契約一覧を兼ねるため |
| 契約詳細 | 満期詳細が契約詳細を兼ねるため |

## 3. 主導線

### 3-1. 日常業務の基本導線

```text
ログイン
  ↓
ダッシュボード
  ↓
満期一覧
  ↓
満期詳細
```

営業担当が最も多く使う導線である。
ダッシュボードは入口、満期一覧は対象抽出、満期詳細は実務処理の中心という役割分担にする。

### 3-2. 顧客確認の導線

```text
ダッシュボード
  ↓
顧客一覧
  ↓
顧客詳細
```

顧客起点で連絡先や保有契約を確認したい場合の導線である。

### 3-3. 満期詳細からの補助導線

```text
満期詳細
  ├─ 顧客詳細
  └─ 事故案件一覧 または 事故案件詳細
```

満期詳細は契約実務の中心画面だが、必要に応じて顧客文脈や事故案件を補助的に確認できるようにする。
ただし、契約専用詳細画面には遷移しない。

## 4. 画面間の関係

### 4-1. ダッシュボード起点

* ダッシュボード → 満期一覧
* ダッシュボード → 顧客一覧
* ダッシュボード → 実績管理一覧
* ダッシュボード → 事故案件一覧
* ダッシュボード → テナント設定

### 4-2. 満期業務

* 満期一覧 → 満期詳細
* 満期詳細 → 顧客詳細

満期一覧と満期詳細が、そのまま契約一覧と契約詳細の役割を兼ねる。
そのため、契約専用画面への遷移は存在しない。

### 4-3. 顧客業務

* 顧客一覧 → 顧客詳細
* 顧客詳細 → 満期詳細

顧客詳細では、その顧客に紐づく契約群を確認できる。
契約の個別確認や更新が必要な場合は、対象契約の満期詳細へ進む。

### 4-4. 事故業務

* 事故案件一覧 → 事故案件詳細
* 事故案件詳細 → 顧客詳細（必要に応じて）

## 5. 設計方針

### 5-1. 画面を増やしすぎない

同じ実体を別画面に分けると、利用者が迷い、説明も複雑になる。
このため、契約と満期は別画面に分けず、同一画面上で扱う。

### 5-2. 一覧で探し、詳細で処理する

基本は「一覧で探す → 詳細で判断・更新する」という流れに統一する。
一覧画面は対象抽出、詳細画面は判断と記録に集中させる。

### 5-3. 顧客は独立した確認軸として持つ

契約処理の途中でも、顧客全体の文脈を確認したい場面は多い。
そのため、顧客一覧・顧客詳細は独立画面として維持する。

### 5-4. ダッシュボードは補助入口であり、処理画面ではない

ダッシュボードは全体状況をざっと見て、必要な一覧へ入るための入口である。
実際の登録・更新・確認作業は各一覧画面および詳細画面で行う。

## 6. 今回の整理で確定したこと

* 満期一覧は契約一覧を兼ねる
* 満期詳細は契約詳細を兼ねる
* 契約一覧は独立画面として作らない
* 契約詳細は独立画面として作らない
* 顧客一覧は正式画面として持つ
* 顧客詳細は正式画面として持つ

## 7. 補足

本ドキュメントは画面の役割と導線を定める上位方針である。
各画面の項目、ボタン、タブ、入力仕様、権限制御の詳細は個別画面設計書に記載する。
