# t_activity.purpose_type マスタ化対応方針

## 1. 問題

`t_activity.purpose_type` は当初 VARCHAR 自由入力として設計されていたが、
`m_activity_purpose_type` マスタが後から追加された（Phase 設定A）。

画面仕様（`activity-detail.md` セクション4-1）では
「purpose_type はマスタ選択のみ、フリーテキスト入力禁止」と定めているが、
マスタが存在しない状態でシステムが先行稼働した場合、
自由入力値または NULL が蓄積される可能性がある。

## 2. 方針決定

### 2-1. 新規登録・更新時の扱い

`m_activity_purpose_type` が1件以上存在する場合は、マスタ選択のみを許容する。
マスタが0件の場合は `purpose_type` を NULL として保存し、バリデーションエラーとしない。

これにより、マスタ未設定でも活動登録が止まらない。

### 2-2. 表示時の扱い

`purpose_type` の値を表示する際、`m_activity_purpose_type.code` と照合する。

| 状態 | 表示 |
|---|---|
| マスタに存在する `code` | `m_activity_purpose_type.label` を表示 |
| マスタに存在しない値（自由入力値） | 値をそのまま表示（グレーアウト推奨） |
| NULL | 「未設定」と表示 |

### 2-3. マスタ化前に蓄積された自由入力値の扱い

マスタ設定前に蓄積された自由入力値については**一括変換は行わない**。

理由：どの自由入力値をどのマスタ `code` に対応させるかは
人間の判断が必要であり、自動変換は誤変換リスクが高い。

担当者が活動詳細を開いた際に、マスタ外の値がグレーアウト表示されるため、
必要であれば担当者が手動で選び直す運用とする。

### 2-4. マスタの初期シード

`m_activity_purpose_type` テーブルのシードデータは DDL ファイル内に定義済みである。
（`renewal` / `new_business` / `cross_sell` / `accident` / `follow_up` / `admin` / `meeting` / `training` / `other`）

テナント設定の「用件区分マスタ」セクションから追加・編集・無効化が可能になる（Phase 設定A以降）。

### 2-5. バリデーションの実装方針

活動登録・更新時のバリデーションは以下のとおり。

```
purpose_type が NULL または空文字 → そのまま保存（エラーなし）
purpose_type が非NULL の場合 → m_activity_purpose_type に is_active=1 のレコードとして存在するか確認
  存在する → 保存OK
  存在しない → バリデーションエラー「用件区分が無効です。再度選択してください。」
```

既存レコードに無効化済みの `code` が設定されている場合、
詳細画面の表示はグレーアウトで許容するが、更新保存時には有効なコードへの変更を求める。

## 3. 今後の拡張上の注意

`purpose_type` は VARCHAR(50) のフリーテキストカラムのままで、
`m_activity_purpose_type` との整合性は DB 制約ではなくアプリ層で保証する設計とする。

マスタの `code` が変更された場合、既存レコードの `purpose_type` 値は自動追従しない。
コードの変更は原則として禁止とし、表示名（`label`）の変更のみを許容する。
廃止したい区分は `is_active = 0` に変更して無効化し、物理削除は行わない。
