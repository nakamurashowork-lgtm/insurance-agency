# 正式導線ポリシー

## 1. この文書の目的

この文書は、どの業務画面を正式導線として扱い、どの画面を通常導線から外すかを定義する。

主題はファイル構成ではなく、利用者がどの画面から日常業務に入り、どの画面を基準に関連画面へ進むべきかである。

この文書の役割は、画面の存在有無を説明することではない。  
通常運用で利用者を迷わせないために、何を主導線とし、何を補助導線とし、何を通常導線から外すかを決めることにある。

## 2. 正式導線の基本原則

正式導線は、次の原則に基づいて定義する。

- 正規スキーマに基づく正式業務を扱うこと
- 利用者が日常業務で繰り返し使う入口であること
- 一覧から詳細へ自然に遷移できる、または一覧中心で利用目的が完結すること
- 関連情報への往復が可能であること
- 非正規画面、暫定実装、移行中機能を前提にしないこと
- 同じ業務に対して二重の主導線を作らないこと

この文書では、業務概念として存在することと、通常運用の主導線に載せることを明確に分けて扱う。

## 3. 現時点の正式な主要入口

現時点で、通常運用上の主要入口とする画面は以下である。

- ホーム
- 満期案件一覧
- 顧客一覧
- 事故案件一覧
- 成績管理
- 活動一覧（営業活動管理）

以下を追加済み。

- 見込案件一覧（営業案件管理）

これ以外の画面は、補助入口、関連画面、または管理画面として位置づける。

ホームは正式な入口画面であり、主要業務へ入るための起点とする。

なお、ナビゲーション上の表示名としては「満期一覧」を用いる場合があるが、正式な業務概念としては満期案件一覧を指す。

## 4. main nav に出してよい画面の基準

main nav に出してよいのは、複数利用者に共通し、継続利用される正式業務の入口に限る。

main nav 掲載の条件は以下である。

- 日常業務の入口として意味が明確である
- 画面名だけで業務目的が伝わる
- 正規画面群への主導線として機能する
- 一時的な補助機能や移行中機能ではない
- 一覧または管理中心の入口として説明できる

この前提では、ホーム、満期案件一覧、顧客一覧、事故案件一覧、成績管理、活動一覧（営業活動）、見込案件一覧（営業案件）は main nav に残す。

一方で、以下は main nav の対象としない。

- 顧客詳細
- 契約詳細
- 満期案件詳細
- 変更履歴確認
- テナント設定
- 通知設定相当の管理機能
- 非正規画面、旧画面、暫定画面

## 5. 入口画面としてのホーム

ホームは、全体状況を確認し、主要業務へ進むための正式な入口画面である。

ホームの役割は以下に限定する。

- 利用者がその日に確認すべき要約情報を示す
- 正式な主要業務への導線をまとめる
- 成績や案件の要約値を表示する
- 注意喚起やお知らせを補助的に表示する

ホームは、各業務画面の代替処理画面にしてはならない。

したがって、ホーム上の日常業務カードは、正式な主要入口と整合していなければならない。  
通常導線に含めない画面を、ホームの日常業務カードや主要ボタンから一般利用者へ誘導してはならない。

**業務入口カードの構成（確定）**

ホームの業務入口カードは「満期業務・事故案件・見込管理」の3枚構成とする。

顧客一覧は業務入口カードには含めない。  
顧客一覧はナビバーに常時表示されているため、ホームから外しても導線の問題は起きない。  
「顧客を探す」は辞書的な行動であり、毎日の業務入口には馴染まない。

営業活動カードは業務入口エリアではなく、成績サマリ下の補助エリアに配置する。  
営業活動カードはトグルを持たず、常にログインユーザー自身の値を表示する。

## 6. 満期業務の主導線

満期業務の主導線は、満期案件一覧と満期案件詳細を中心とする満期案件系である。

主導線の定義は次の通り。

- 入口は満期案件一覧とする
- 一覧で対象案件を絞り込み、進捗状況を把握する
- 詳細で個別案件を処理する
- 詳細から契約詳細、顧客詳細、必要な補助情報へ進む

満期業務の説明、通知条件、コメント運用、関連データ確認導線は、この主導線を基準に揃える。

一覧から詳細へ進む流れを主導線とし、一覧からの直接編集は補助導線として扱う。

## 7. 成績管理の位置づけ

成績管理は、満期業務とは別軸の正式業務であるため、主導線として残す。

成績管理の役割は以下である。

- 計上成績を一覧で確認する
- 条件を絞って成績傾向を把握する
- 登録、更新、削除、CSV取込を行う
- ホームとは補助的に連携する

したがって、成績管理は非正規画面ではなく正式画面として扱う。

## 8. 非正規画面を通常導線に出さない方針

非正規画面や移行中画面は、通常導線に出さない。

理由は、利用者に二重導線を認識させると、どちらが正式なのかが曖昧になり、運用が分散するためである。

非正規画面の扱いは以下を原則とする。

- main nav には出さない
- ホームの日常業務カードからも通常は遷移させない
- 一般利用者の通常導線では到達させない
- 管理者が必要時に参照する場合でも、正式導線ではないことが分かる扱いに留める

## 9. 営業活動管理の位置づけ（正式導線として追加）

営業活動管理は、顧客への訪問・電話・メールなどの活動記録を管理する業務であり、**満期業務・顧客業務・成績管理と並ぶ正式業務**として扱う。

**営業活動管理が正式導線である根拠は以下の通り。**

- 日常業務で繰り返し使う入口であること（営業担当が毎日記録する）
- 顧客起点であり、既存の顧客業務・満期業務・事故業務と連携が必要なこと
- 正規スキーマ（t_activity）に基づく正式業務であること
- 活動の集約として日報ビューが成立するため、独立画面群として完結すること

したがって、現時点での営業活動管理に関する通常導線の原則は以下の通りである。

- main nav には「**営業活動**」として掲載し、リンク先は `activity/daily`（当日の日報ビュー）とする
- 活動一覧（`activity/list`）は **検索・集計用**。日報ビューの「過去の活動を検索」リンクから到達する
- 活動一覧・活動登録・活動詳細・日報ビューは **一般利用者向けの標準業務フローに含まれる**
- **何の制限もなく全ログインユーザーがアクセス可能**

**activity/new と activity/detail の責務分離**

活動登録と活動詳細は、画面名・URLが別であり、責務を明確に分ける。

| 画面 | ルート | 責務 |
|------|--------|------|
| 活動登録 | `activity/new` | **新規登録のみ**。フォームを空で開始し、登録後は activity/daily に戻る |
| 活動詳細 | `activity/detail` | **既存活動の確認・編集のみ**。活動IDと `?from=` を受け取り、既存データを表示して更新・削除を行う |

活動登録画面から既存データを参照・編集してはならない。
活動詳細画面を新規登録の起点として使ってはならない。

### 9-1. 活動画面の責務分離

| 画面 | ルート | 主な用途 | 主要入口 |
|------|--------|----------|---------|
| 日報ビュー | `activity/daily` | 当日の活動確認・登録・日報コメント | main nav「営業活動」、ホームカード |
| 活動一覧 | `activity/list` | 期間・担当者・顧客名で横断検索・集計 | 日報ビュー「過去の活動を検索」|
| 活動登録 | `activity/new` | 新規登録専用 | 日報ビューのダイアログ |
| 活動詳細 | `activity/detail` | 既存活動の確認・編集・削除 | 日報/一覧/顧客詳細の活動リンク |

**日報ビューの位置づけ**

日報ビュー（`activity/daily`）は日常の活動入口であり、main nav の主要ターゲットである。
その日の活動を集約表示し、日報コメントの入力・更新・活動登録ダイアログを持つ。
ヘッダーに「過去の活動を検索」リンクを配置し、活動一覧へ誘導する。

**日報提出フローの位置づけ（現状: UI 非表示）**

日報提出機能は本フェーズでは UI から削除されている。
承認フロー（上司確認 → 承認/差し戻し）が未実装のため、提出の業務的意味がない。
DDL（`t_daily_report.is_submitted`, `submitted_at`）と Repository（`submit()` メソッド）は
将来の承認フロー実装に備えて残してある。
復活させる際は、承認者側の画面と権限制御を合わせて実装する必要がある。

## 10. 見込案件管理の位置づけ（正式導線として追加済み）

見込案件管理は、顧客への新規提案・クロスセル提案などの営業パイプラインを管理する業務であり、**活動管理・顧客管理と連携する正式業務**として扱う。

**見込案件管理が必要な根拠は以下の通り。**

- Excel「日報」の「見込み」シートで月次管理されていた業務をWebに移行するものである
- 活動記録（t_activity）が「何の案件に向けた活動か」を紐づけるために t_sales_case が必要である
- Phase A/B で sales_case_id を NULL 固定のまま運用開始すると、活動と見込案件の二重管理が恒久化する

**通常導線の原則は以下の通り。**

- 見込案件一覧（`sales-case/list`）は **main nav の正式な主要入口に掲載済み**（「見込案件」として）
- 見込案件一覧・登録・詳細は **一般利用者向けの標準業務フローに含まれる**
- 活動登録画面の sales_case_id プルダウンを有効化し、活動と見込案件を紐づけられるようにする
- 顧客詳細から「この顧客の見込案件一覧」を参照できるようにする

## 11. 事故案件の位置づけ（正式導線への昇格完了）

事故案件は、正規スキーマ上は正式な業務概念である。

**現在、事故案件業務は通常運用の正式な主導線として扱っており、補助導線ではない。**

事故対応は、顧客からの連絡を受けて営業担当や事務担当が随時確認・進捗管理・対応記録を行う日常業務であり、満期案件一覧や成績管理と並ぶ正式導線として機能している。

したがって、現時点での事故案件に関する通常導線の原則は以下の通りである。

- 事故案件一覧・詳細は **main nav の正式な主要入口に掲載** される（「事故管理」として）
- 事故案件は **ホームの日常業務カードに含まれる** （事故対応導線として）
- 事故案件は **一般利用者向けの標準業務フローに含まれる** （営業担当・内勤担当の日常業務）
- 事故案件詳細は **一覧から入る通常の詳細導線として扱う** （満期詳細などと同格）
- 何の制限もなく **全ログインユーザーがアクセス可能** である

事故案件の正式導線化に伴い、本方針、overview、screen-map、個別画面設計、implementation-plan の整合性を保つ必要がある。  
個別画面設計でこの方針と矛盾する制限や補助位置づけを導入してはならない。

## 12. 詳細画面からの導線原則

詳細画面は、関連データへの往復導線を担う。

原則は以下である。

- 顧客詳細は、顧客情報の確認を中心とする補助画面とする
- 契約詳細は、契約条件確認を中心とする補助画面とする
- 満期案件詳細は、案件文脈の確認を行う中核画面とする
- 詳細画面から関連画面へ進めるようにする
- 詳細画面から非正規画面へ誘導しない

具体的には、以下の関係を基本とする。

- 満期案件詳細から契約詳細へ進める
- 満期案件詳細から顧客詳細へ進める
- 契約詳細から関連顧客へ進める
- 契約詳細から関連する満期案件文脈へ戻れるようにする
- 顧客詳細から必要に応じて関連契約へ進める

## 13. 変更履歴の扱い

変更履歴は正式機能である。  
ただし、この文書では、変更履歴を独立補助画面として見せるか、各詳細画面内の補助領域として見せるかまでは確定しない。

この文書で確定するのは次の点である。

- 変更履歴は主導線の入口ではない
- 変更履歴は詳細画面の文脈で利用する補助機能である
- 顧客、契約、満期案件、コメントなどの対象ごとに確認できる必要がある
- 変更履歴の見せ方は screen-map と個別画面設計で統一する

したがって、個別画面設計で変更履歴の見せ方を定義する場合でも、この文書の前提を壊してはならない。

## 14. 設定系画面の扱い

設定系画面は、通常業務を支える管理機能であり、日常業務の主導線とは分けて扱う。

原則は以下の通り。

- 一般利用者向けの主要入口にはしない
- main nav の正式入口にはしない
- 管理者向けの補助導線またはヘッダーメニュー配下で扱う
- 権限差分は、業務主導線ではなく管理機能の公開範囲に用いる

通知設定を独立画面として扱うか、テナント設定内の1機能として扱うかは、screen-map と個別設計で統一する。  
ただし、どちらの構成を採る場合でも、設定系である以上、通常業務の主導線とは分ける。

## 15. 権限による入口差分

権限差分は、業務主導線を分けるためではなく、設定系や管理機能の公開範囲を制御するために使う。

- 一般ユーザーは正式業務画面に集中できる構成とする
- admin と system_admin は、テナント設定や通知設定相当の管理機能へ進める
- 権限差分は業務主導線ではなく管理機能の公開範囲に用いる

通常利用者と管理者で、まったく別の業務主導線を持たせない。

## 16. 運用上の判断基準

画面を正式導線に残すか迷う場合は、次の問いで判断する。

1. 正規スキーマ上の正式業務を扱っているか
2. 日常業務の最初の入口として説明できるか
3. 他の正式画面と役割が重複していないか
4. 利用者がその画面を使う理由を画面名だけで理解できるか
5. 非正規画面や暫定実装を前提にしていないか

この問いに明確に答えられない画面は、通常導線に置かない。
