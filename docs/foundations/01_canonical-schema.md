# 正規スキーマ

## 1. 位置づけ

この文書は、tenant DB 側の正規スキーマを、業務責務の観点で整理するものである。  
DDL の転記ではなく、それぞれのテーブルが何を保持し、どの業務概念を担うかを説明する。

ここで扱うのは、あくまで tenant DB 側の正式スキーマである。  
画面導線、画面レイアウト、実装状況、旧画面の扱いは主題にしない。

この文書の目的は、業務データの責務分担を明確にし、後続の画面設計や実装判断で、どの概念をどのテーブルで持つべきか迷わない状態をつくることである。

## 2. この文書で定義すること / しないこと

この文書で定義することは以下である。

- tenant DB 側で保持する業務概念
- 各テーブルの責務
- テーブル同士の大まかな関係
- どの業務概念をどのテーブル群で支えるか

この文書で定義しないことは以下である。

- どの画面を正式導線に載せるか
- 変更履歴を独立画面で見せるか、詳細画面内で見せるか
- コメントや通知をどの画面でどう表示するか
- admin DB や共通設定側で持つ情報の詳細構造

つまり、この文書は「何をどこで保持するか」を定義する文書であり、「どう見せるか」は後続文書の責務とする。

## 4. 業務概念ごとの責務

### 4-1. 顧客

#### m_customer

顧客という業務主体を保持する中核テーブルである。  
個人・法人の別、名称、連絡先の基礎情報、担当者、状態など、業務上の参照起点になる情報を持つ。

このテーブルの責務は、契約や案件の親となる顧客の基本単位を、安定して管理することである。

### 4-2. 契約

#### t_contract

保険契約そのものを表すテーブルである。  
契約者となる顧客、証券番号、保険種類、契約日、始期日、終期日、保険料、担当者、状態などを保持する。

**保険会社カラムについて**  
本システムは損保ジャパン専用のため、保険会社カラム（`insurer_name`）は保持しない。全ての契約は損保ジャパンのものとして扱う。もし将来、他社保険を扱う必要が生じた場合は、別途マスタ化を検討する。

**年度別複数レコード設計**  
同一証券番号でも終期日（`policy_end_date`）が異なれば別の契約レコードとして保持する。  
UNIQUE KEY は `(policy_no, policy_end_date)` の複合制約であり、同一証券番号・同一終期の重複のみを防ぐ。  
SJNET CSV 取込のたびに新年度分を新規 INSERT し、旧年度の契約は履歴として残す。  
満期案件や活動履歴は各年度の `contract_id` に紐づく。

### 4-3. 満期案件

#### t_renewal_case

契約の更新対応や満期前後の進捗管理を行う案件テーブルである。  
満期日、状態、担当者、更改方法、手続方法、対応完了日、メモなどを保持する。

このテーブルの責務は、契約そのものではなく、契約に対して発生する満期対応業務を管理することである。  
通知対象判定もこの案件の状態を基準に行う。

#### m_renewal_reminder_phase

満期通知のフェーズ定義を持つマスタである。  
どの時点でどの通知を行うかという、業務上の段階を表現する。

このテーブルの責務は、満期案件に対する通知時期の意味づけを固定し、案件ごとの通知判定基準をぶらさないことにある。

### 4-4. 事故案件

#### t_accident_case

事故や損害に関する案件を保持するテーブルである。  
契約や顧客と関連しつつ、満期案件とは別の業務文脈で進捗を管理する。

このテーブルの責務は、事故対応という別系統の案件を、契約更新業務とは独立した案件単位として保持することである。

### 4-5. 活動

#### t_activity

担当者が行った接触、対応、フォロー、記録などを蓄積する活動テーブルである。  
必ず顧客に紐づき、必要に応じて契約、満期案件、事故案件のいずれかにも関連づけられる。  
`sales_case_id` カラムで営業案件（`t_sales_case`）への紐づけも保持できるが、Phase C まで UI からは設定しない。Phase A/B では保存時に常に NULL 固定とする。

このテーブルの責務は、案件の状態変化とは別に、実際の行動履歴を時系列で持てるようにすることである。

### 4-5a. 日報コメント

#### t_daily_report

担当者×日付の単位で日報コメントを保持するテーブルである。  
UNIQUE KEY `(report_date, staff_user_id)` により、1スタッフ1日につき1件の記録を保証する。  
`is_submitted` は日報提出フロー向けのフラグであり、提出操作時に 1 へ更新する。提出済みの場合は `submitted_at` を合わせて記録する。未提出時は 0 のまま保持する。

このテーブルの責務は、活動記録そのもの（`t_activity`）とは独立して、担当者が日単位で付与するコメントを保持することである。  
活動記録の代替ではなく、日次集約ビューの補助記録として位置づける。

### 4-5b. 営業案件

#### t_sales_case

顧客起点の営業パイプラインを保持するテーブルである。  
顧客や契約に紐づく案件名、案件種別、ステータス、見込み度、想定保険料、契約予定月などを持ち、営業活動の管理基盤になる。  
`t_activity` の `sales_case_id` から参照されることで、個々の活動と案件を結びつける。

このテーブルの責務は、顧客への営業進捗を案件単位で管理し、活動記録の文脈を補完することである。  
**Phase C まで UI 実装は行わない。** DDL は追加済みだが、Phase A/B では画面・操作の対象外とする。

### 4-6. コメント

#### t_case_comment

案件に付随する人間向けの補足情報を保持するテーブルである。  
満期案件と事故案件の両方に対して、共通のコメント基盤として使う。

このテーブルの責務は、案件処理に関するメモや引継ぎ事項を、案件本体の正式属性とは分けて保持することである。  
削除状態はこのテーブルで管理し、変更経緯の追跡は変更履歴テーブル側で担保する。

### 4-7. 成績

#### t_sales_performance

営業成績や計上成績を保持するテーブルである。  
計上日、契約者名、保険種類、種目、保険料、精算月、担当者などを持ち、集計と一覧確認の基盤になる。

このテーブルの責務は、契約管理とは別軸で、成績把握や月次比較に必要な成績データを保持することである。

`premium_amount` は円単位の整数値で保存する（DECIMAL(12,0) に対応）。小数点以下は扱わない。CSV取込時も同様の単位で保存する。

**保険会社カラムについて**  
損保・生保とも、保険会社カラム（`insurer_name`）は保持しない。損保は損保ジャパン固定。生保は会社名が必要な場合、商品名（`product_type`）に含める運用とする（例: "損保ジャパンひまわり生命 終身保険"）。

### 4-8. 変更履歴

#### t_audit_event

業務データに対して発生した操作イベントの単位を保持するテーブルである。  
誰が、いつ、どの対象に対して、どの種別の変更を行ったかを記録する。

このテーブルの責務は、変更履歴の親イベントを保持し、業務オブジェクト単位の履歴追跡を可能にすることである。

#### t_audit_event_detail

監査イベントに紐づく項目単位の変更内容を保持するテーブルである。  
変更前後の値や表示順を持ち、何がどう変わったかを読めるようにする。

このテーブルの責務は、親イベントだけでは表し切れない変更差分を、項目単位で分解して保持することである。

なお、変更履歴を独立画面で見せるか、各詳細画面の補助領域として見せるかは、この文書では定義しない。  
この文書では、変更履歴データをどのテーブルで保持するかだけを定義する。

### 4-9. 通知実行

#### t_notification_run

通知実行の単位を保持するテーブルである。  
いつ、どの種別の通知処理を実行し、全体としてどういう結果になったかを記録する。

このテーブルの責務は、通知処理そのものの実行履歴を残すことである。

#### t_notification_delivery

通知実行の結果として、どの対象へどの配信結果になったかを保持するテーブルである。  
満期案件と事故案件のどちらに対する通知かを識別しつつ、個別配信単位の結果を持つ。

このテーブルの責務は、通知が実際にどう処理されたかを、個別対象単位で確認できるようにすることである。

ここで扱うのは、あくまで tenant DB 上の通知実行と配信結果である。  
通知先そのものの共通設定、通知ルートの管理、外部サービス接続設定の全体構造は、この文書の対象外とする。

### 4-10. 事故通知条件

#### t_accident_reminder_rule

事故案件に対する通知ルールの本体を持つテーブルである。  
事故案件でどのような通知条件を採用するかを管理する。

#### t_accident_reminder_rule_weekday

事故通知ルールのうち、曜日など繰り返し条件を分離して持つテーブルである。

この2テーブルの責務は、事故案件の通知条件を構造化して保持することである。

### 4-11. SJNET import

#### t_sjnet_import_batch

SJNET からの取込実行単位を保持するテーブルである。  
いつ、どのファイルや条件で取り込んだか、全体としてどうだったかを表す。

このテーブルの責務は、外部取込を再現可能かつ検証可能な形で残すことである。

#### t_sjnet_import_row

取込バッチに含まれる行単位の結果を保持するテーブルである。  
元データ、行番号、紐づいた契約や満期案件、行ごとの判定結果を保持する。

`t_sjnet_import_batch` は取込実行の単位を保持し、`t_sjnet_import_row` はその実行に含まれる行ごとの判定結果を保持する。  
両者を分けることで、取込全体の再現性と、個別行の検証可能性を両立する。

## 5. テーブル間の関係

正規スキーマ上の主な関係は次の通りである。

- 顧客は契約の親になる
- 顧客は複数の連絡先を持てる
- 契約は満期案件の親になる
- 活動は顧客を必須の起点とし、必要に応じて契約、満期案件、事故案件、営業案件に紐づく（営業案件との紐づけは Phase C から）
- 日報コメントは担当者×日付で1件、活動記録とは独立して保持する
- 営業案件は顧客を起点とし、必要に応じて契約に紐づく
- 満期案件や事故案件にはコメントが紐づく
- 顧客、契約、案件、活動、コメント、成績などの主要オブジェクトは、業務オブジェクト単位で変更履歴記録の対象になり得る
- 通知実行は run と delivery に分かれ、delivery は満期案件または事故案件に紐づく
- SJNET import は batch と row に分かれ、row 単位で契約や満期案件との対応結果を持つ

## 6. どの業務概念をどのテーブルで持つか

| 業務概念 | 主テーブル | 補助テーブル | 説明 |
| --- | --- | --- | --- |
| 顧客 | m_customer | なし | 顧客の基本情報を保持する |
| 契約 | t_contract | なし | 契約情報の正式な保持先 |
| 満期案件 | t_renewal_case | m_renewal_reminder_phase | 契約に対する更新対応業務を保持 |
| 事故案件 | t_accident_case | t_accident_reminder_rule, t_accident_reminder_rule_weekday | 事故対応業務を保持 |
| 活動 | t_activity | なし | 日々の対応履歴を保持 |
| 日報コメント | t_daily_report | なし | 担当者×日付で1件の日報コメントを保持（活動記録の補助） |
| 営業案件 | t_sales_case | なし | 顧客起点の営業パイプラインを保持（Phase C から利用） |
| コメント | t_case_comment | なし | 案件に紐づく補足記録 |
| 成績 | t_sales_performance | なし | 営業成績・計上成績を保持 |
| 変更履歴 | t_audit_event | t_audit_event_detail | 業務オブジェクト単位の変更イベントと差分を保持する |
| 通知実行 | t_notification_run | t_notification_delivery | 通知実行と配信結果を保持 |
| SJNET import | t_sjnet_import_batch | t_sjnet_import_row | 外部取込の履歴を保持 |

## 7. 設計上の補足

- 契約は顧客に従属するが、満期案件は契約に対して発生する別の業務単位である
- 事故案件も正規スキーマ上の正式概念であるが、この文書は画面上の主導線を定義しない
- コメントは案件本体の正式属性ではなく、運用上の補助記録として独立させる
- 変更履歴は表示方法ではなく、どの業務オブジェクトに対する変更を記録するかを基準に整理する
- 通知は設定、実行、配信結果を分けて考える必要がある。この文書では tenant DB 側の実行・結果・案件別条件を対象とし、共通設定側は別論点として扱う
- `t_daily_report` は活動記録（`t_activity`）の集約ビューに対する補助コメントであり、活動そのものを保持するテーブルではない
- `t_sales_case` の DDL は追加済みだが、Phase C まで UI から利用しない。`t_activity.sales_case_id` との紐づけも Phase C から有効化する
- この文書は tenant DB 側の正規スキーマを対象とするため、画面設計書に書くべき導線ルールは持ち込まない
