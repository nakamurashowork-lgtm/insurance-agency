# 画面マップ

## 1. 目的

本システムの画面構成と主要導線を整理する。

画面は増やしすぎず、営業担当が日常業務で迷わず使えることを優先する。
特に、満期業務では **満期一覧 = 契約一覧**、**満期詳細 = 契約詳細** として扱い、契約専用の一覧画面・詳細画面は別途設けない。

また、顧客軸での確認や連絡履歴の把握が必要なため、顧客一覧・顧客詳細は正式画面として持つ。

### 2-2. 設けない画面

以下は独立画面としては持たない。

| 画面名 | 理由 |
|--------|------|
| 契約一覧 | 満期一覧が契約一覧を兼ねるため |
| 契約詳細 | 満期詳細が契約詳細を兼ねるため |

## 3. 主導線

### 3-1. 日常業務の基本導線

```text
ログイン
  ↓
ホーム
  ↓
満期一覧
  ↓
満期詳細
```

営業担当が最も多く使う導線である。
ホームは入口、満期一覧は対象抽出、満期詳細は実務処理の中心という役割分担にする。

### 3-2. 顧客確認の導線

```text
ホーム
  ↓
顧客一覧
  ↓
顧客詳細
```

顧客起点で連絡先や保有契約を確認したい場合の導線である。

### 3-3. 事故業務の導線

```text
ホーム
  ↓
事故案件一覧
  ↓
事故案件詳細
```

営業担当や事務担当が日常的に使う事故対応の導線である。

### 3-4. 実績業務の導線

```text
ホーム
  ↓
実績管理一覧
  ↓
実績詳細
```

実績管理は一覧で対象を探し、詳細で1件確認と更新を行う導線である。

### 3-5. 満期詳細からの補助導線

```text
満期詳細
  ├─ 顧客詳細
  └─ 事故案件一覧 または 事故案件詳細
```

満期詳細は契約実務の中心画面だが、必要に応じて顧客文脈や事故案件を補助的に確認できるようにする。
ただし、契約専用詳細画面には遷移しない。

### 3-6. 営業活動の導線

```text
main nav「営業日報」→ activity/daily（当日の日報ビュー）

ホーム → 活動一覧（activity/list）
  ├─ 活動登録（新規専用）→ 登録後 活動一覧に戻る
  ├─ 活動詳細（既存の確認・編集専用）
  └─ 日報ビュー（日付リンク → その日の活動まとめ + コメント入力 + 日報提出）
```

顧客詳細からも活動登録・活動詳細へ進める（customer_id を引き継ぐ）。

**activity/new と activity/detail の責務分離**

| 画面 | ルート | 責務 |
|------|--------|------|
| 活動登録 | `activity/new` | 新規登録のみ。登録後は activity/list に戻る |
| 活動詳細 | `activity/detail` | 既存活動の確認・編集・削除のみ。`?id=` で活動IDを受け取る |

**日報提出フロー**

日報ビュー（`activity/daily`）から `POST activity/submit` で提出する。
提出後は is_submitted=1 / submitted_at=NOW() が記録される。提出の取り消しはできない。
管理者は活動一覧の「日報提出状態」フィルタで未提出担当者を把握できる。

### 3-7. 見込案件の導線

```text
見込案件一覧
  ├─ 見込案件登録（新規専用）→ 登録後 見込案件一覧に戻る
  └─ 見込案件詳細（既存の確認・編集・削除専用）
```

顧客詳細からも「この顧客の見込案件一覧」を参照できる。
活動登録画面で sales_case_id を見込案件から選択できる。

## 4. 画面間の関係

### 4-1. ホーム起点

業務入口カード経由の導線:

- ホーム → 満期一覧（業務入口カード「満期業務」）
- ホーム → 事故案件一覧（業務入口カード「事故案件」）
- ホーム → 見込案件一覧（業務入口カード「見込管理」）
- ホーム → 活動一覧（下部「営業活動」カード）
- ホーム → 実績管理一覧（下部「実績管理」カード、および実績サマリのリンク）

ナビバー経由の導線（業務入口カードには含めない）:

- ナビバー → 顧客一覧（「顧客を探す」は辞書的な行動のためナビバーに委ねる）

管理者のみ:

- ホーム → テナント設定（管理セクション）

### 4-2. 実績業務

- 実績管理一覧 → 実績詳細

### 4-3. 満期業務

- 満期一覧 → 満期詳細
- 満期詳細 → 顧客詳細

満期一覧と満期詳細が、そのまま契約一覧と契約詳細の役割を兼ねる。
そのため、契約専用画面への遷移は存在しない。

### 4-4. 顧客業務

- 顧客一覧 → 顧客詳細
- 顧客詳細 → 満期詳細
- 顧客詳細 → 見込案件詳細

顧客詳細では、その顧客に紐づく契約群を確認できる。
契約の個別確認や更新が必要な場合は、対象契約の満期詳細へ進む。

### 4-5. 事故業務

- 事故案件一覧 → 事故案件詳細
- 事故案件詳細 → 顧客詳細（必要に応じて）

### 4-6. 営業活動業務

- 活動一覧 → 活動登録（新規専用）
- 活動一覧 → 活動詳細（既存の確認・編集専用）
- 活動一覧 → 日報ビュー（日付リンク経由）
- 日報ビュー → 日報提出（POST activity/submit。本人のみ）
- 顧客詳細 → 活動登録（customer_id 引き継ぎ）
- 顧客詳細 → 活動詳細（顧客に紐づく活動リンク）

### 4-7. 見込案件業務

- 見込案件一覧 → 見込案件登録（新規専用）
- 見込案件一覧 → 見込案件詳細（既存の確認・編集専用）
- 顧客詳細 → 見込案件一覧（顧客に紐づく見込案件）
- 活動登録 → 見込案件選択（sales_case_id プルダウン実装済み）

## 5. 設計方針

### 5-1. 画面を増やしすぎない

同じ実体を別画面に分けると、利用者が迷い、説明も複雑になる。
このため、契約と満期は別画面に分けず、同一画面上で扱う。

### 5-2. 一覧で探し、詳細で処理する

基本は「一覧で探す → 詳細で判断・更新する」という流れに統一する。
一覧画面は対象抽出、詳細画面は判断と記録に集中させる。

### 5-3. 顧客は独立した確認軸として持つ

契約処理の途中でも、顧客全体の文脈を確認したい場面は多い。
そのため、顧客一覧・顧客詳細は独立画面として維持する。

### 5-4. ホームは入口画面であり、処理画面ではない

ホームは全体状況をざっと見て、必要な一覧へ入るための入口である。
実際の登録・更新・確認作業は各一覧画面および詳細画面で行う。

## 6. 今回の整理で確定したこと

- 満期一覧は契約一覧を兼ねる
- 満期詳細は契約詳細を兼ねる
- 契約一覧は独立画面として作らない
- 契約詳細は独立画面として作らない
- 顧客一覧は正式画面として持つ
- 顧客詳細は正式画面として持つ
- 事故案件一覧は main nav とホームの日常業務に含める
- main nav は「営業日報」として `activity/daily` を入口とする（`activity/list` はホーム経由）
- 活動登録（activity/new）は新規登録専用画面とし、既存データ編集に使わない
- 活動詳細（activity/detail）は既存活動の確認・編集専用画面とし、新規登録の起点に使わない
- 日報ビュー（activity/daily）は活動一覧の補助画面であり、単独の main nav 入口としない
- 日報ビューから日報提出（POST activity/submit）を行う。提出後は取り消し不可
- 見込案件一覧（sales-case/list）は main nav に追加済み

## 7. 補足

本ドキュメントは画面の役割と導線を定める上位方針である。
各画面の項目、ボタン、タブ、入力仕様、権限制御の詳細は個別画面設計書に記載する。
全画面の共通 UI 作法（一覧/詳細/ダイアログ、ボタン、文言、バッジ）は `docs/policies/06_common-ui-rules.md` を参照する。

### 7-1. 画面設計書と画面 ID の対応について

SCR-ACTIVITY-NEW と SCR-ACTIVITY-DETAIL は、**1 つの画面設計書（`docs/screens/activity-detail.md`）** にまとめて記述している。  
これは、両画面が同一のフォーム構成・入力項目を共有しつつ、登録と編集削除で責務を明確に分けているためである（1 ドキュメントで責務分離を明示した方が対比が明快）。

- 画面設計書の表題: 「活動登録 / 活動詳細（SCR-ACTIVITY-NEW / SCR-ACTIVITY-DETAIL）」
- 責務の分離はドキュメントの第 1 節「責務の分離（重要）」に記載する
- 活動に関する画面設計書は 3 ファイル（activity-list.md / activity-detail.md / activity-daily.md）で 4 画面 ID を網羅する

SCR-SALES-CASE-NEW と SCR-SALES-CASE-DETAIL も同様に、1 つの画面設計書にまとめて記述する。
Phase C-Lite 着手時に `docs/screens/sales-case-detail.md` として作成する。
